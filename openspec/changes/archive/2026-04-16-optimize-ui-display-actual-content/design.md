## Context

当前右侧面板使用标签页布局（"元素列表" / "元素详情"），元素列表中每个卡片仅显示截断的 80 字符预览，用户需要切换到"元素详情"才能查看完整内容和进行编辑。项目使用 React + TypeScript + Zustand store，右侧面板组件结构为：
- `RightPanel.tsx`：标签页容器 + `ElementList` / `ElementDetail`
- `ElementList`：拖拽排序的卡片列表，每张卡片含 `ElementContentPreview`
- `ElementDetail`：选中元素的完整渲染预览 + 源码编辑器

Store（`useAnnotationStore`）管理元素数据和选中状态，逻辑层无需改动。

## Goals / Non-Goals

**Goals:**
- 元素列表中每张卡片直接展示渲染后的实际内容（markdown/HTML/latex）
- 移除"元素详情"标签页，将编辑能力内联至列表卡片
- 保持现有拖拽排序、选中高亮、悬停联动等交互不变
- 选中某元素时，在列表内展开该卡片展示编辑器

**Non-Goals:**
- 不修改后端 API 或数据结构
- 不改变 PDF 预览区域（LeftPanel）的交互逻辑
- 不引入新的第三方依赖（复用已有 react-markdown、CodeMirror、katex 等）
- 不改变 store 结构或状态管理方式

## Decisions

### 1. 卡片折叠/展开态替代详情标签页
- **决策**：元素列表卡片支持两种状态——折叠态（显示渲染后内容预览）和展开态（显示内容编辑器）
- **理由**：用户只需在列表内即可完成查看和编辑，无需切换标签页，减少交互步数
- **备选方案**：保留标签页但增强列表内容——被否决，因为两个标签页本质是同一信息的不同呈现密度，展开卡片更自然

### 2. 使用 Zustand store 中的 `selectedElementId` 控制展开态
- **决策**：复用 store 中已有的 `selectedElementId` 作为卡片展开状态标识，被选中的卡片自动展开
- **理由**：避免新增状态字段，保持 store 简洁
- **备选方案**：新增 `expandedElementId`——被否决，因为当前需求中"选中即展开"的语义清晰

### 3. 折叠态直接渲染实际内容而非截断文本
- **决策**：折叠态使用 `RenderedContent` 组件（已有）完整渲染元素内容，通过 CSS max-height 控制显示区域，而非截断源文本
- **理由**：保留完整渲染质量（markdown 表格、公式等），CSS 溢出处理让用户感知内容为"可展开的完整块"
- **备选方案**：仍用源文本截取更多字符——被否决，因为无法展示表格/公式等结构化内容的渲染效果

### 4. 编辑能力保留在展开态卡片内
- **决策**：展开态卡片复用 `ContentEditor`（CodeMirror）和保存逻辑，从原 `ElementDetail` 迁移而来
- **理由**：复用已有成熟组件，减少重构风险

## Risks / Trade-offs

- **[性能] 多元素同时渲染 markdown/HTML 可能影响列表滚动流畅度** → 折叠态使用简单的 CSS overflow 控制，React Compiler 优化重渲染；必要时添加列表虚拟化和懒渲染（只渲染可视区域内的元素）
- **[可用] 卡片内编辑区域空间有限** → 使用 CodeMirror 的可调节高度或最小高度 200px，配合卡片展开后的完整空间
- **[迁移成本] 用户已习惯两个标签页的工作流** → 折叠态/展开态的区别通过视觉引导（如展开箭头图标、背景色变化）明确传达
